## 策略特点

### 核心创新点

1. **多轮迭代架构**
   - 第1轮：生成初始代码
   - 第2-3轮：根据测试反馈修正
   - 充分利用 max_turns=3 的限制

2. **错误分类处理**
   ```
   Runtime Error → 提示检查边界条件、类型错误
   Time Limit Exceeded → 提示优化算法复杂度
   Wrong Answer → 提示检查逻辑错误、边界情况
   ```

3. **系统提示词设计**
   - 强调"expert competitive programmer"身份
   - 分步骤引导思考（问题分析 → 算法设计 → 代码实现）
   - 明确要求考虑边界条件和复杂度

4. **Token 管理**
   - 监控累积 token 数（限制16k）
   - 超过阈值时保留关键信息
   - 避免上下文污染

### 2. 策略优化方向

根据测试结果，可以考虑以下优化：

**如果成功率较低（<50%）：**
- 增强系统提示词，添加更多解题技巧
- 在第一轮就要求模型先分析问题再编码
- 添加 few-shot 示例（需要权衡 token 开销）

**如果 Runtime Error 较多：**
- 强化边界条件检查的提示
- 要求模型先列出可能的边界情况

**如果 TLE 较多：**
- 在系统提示中更强调时间复杂度
- 要求模型先分析复杂度再编码

**如果 Wrong Answer 较多：**
- 要求模型先用样例验证
- 增加逻辑检查的提示

## 注意事项

1. **API 成本控制**：每题3轮调用，20题×16样本 = 960次调用
2. **时间预算**：每题平均2-3分钟，完整评测需要约1小时
3. **结果稳定性**：temperature=1.0 导致结果有随机性，需要多样本评估
4. **模型限制**：Qwen3-8B 能力有限，复杂题目可能难以通过
